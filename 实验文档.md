# 实验一 椭圆曲线算法
## 实验目的
+ 理解非对称加密算法

## 实验介绍
### 非对称加密算法
公开密钥密码学（英语：Public-key cryptography）也称非对称式密码学（英语：Asymmetric cryptography）是密码学的一种算法，它需要两个密钥，一个是公开密钥，另一个是私有密钥；公钥用作加密，私钥则用作解密。使用公钥把明文加密后所得的密文，只能用相对应的私钥才能解密并得到原本的明文，最初用来加密的公钥不能用作解密。由于加密和解密需要两个不同的密钥，故被称为非对称加密；不同于加密和解密都使用同一个密钥的对称加密。公钥可以公开，可任意向外发布；私钥不可以公开，必须由用户自行严格秘密保管，绝不透过任何途径向任何人提供，也不会透露给被信任的要通信的另一方。

### 陷门函数
所有公钥加密算法的关键在于它们各自都有其独特的陷门函数。陷门函数只能被单向计算，或者至少只能容易地单向计算（使用现代计算机在不到几百万年的时间内）

### ECC算法
ECC是椭圆曲线加密（Elliptic Curve Cryptography）的缩写，在网络通信以及区块链系统中最常用的加密算法之一。一条椭圆曲线就是一组被$ y^2 = x^3 + ax +b $定义的且满足$ 4a^3 + 27b^2 \neq 0 $的点集。

在ECC算法中，陷门函数函数是建立在 “有限域上的二元三次曲线上的点”上 ，组成一个[阿贝尔群](https://zh.wikipedia.org/zh/%E9%98%BF%E8%B4%9D%E5%B0%94%E7%BE%A4) （满足封闭性，结合性，有单位元，有逆元）。

在椭圆曲线上的加法采用一种几何加法的方式。首先，在椭圆曲线上取一点P(Xp,Yp)，再取一点Q(Xq,Yq)，连接P、Q两点作一条直线，这条直线将在椭圆曲线上交于第三点G，过G点作垂直于X轴的直线，将过椭圆曲线另一点R（一般是关于X轴对称的点），R点则被定义为P+Q的结果，既P+Q=R。

一个公钥密码学方案使用的椭圆曲线通过如下变量定义：

+ 确定曲线$ y^2 = x^3 + ax + b $中的a和b
+ 确定有限域需要的素数p
+ 确定起点G坐标(x,y)
+ 确定通过G生成的群阶数n

ECC算法是在有限域Fp定义公式：Q=kP，已知大数k和点P的情况下，很容易求点Q，但是已知的点P、点Q，却很难求得k，这就是经典的离散对数问题，ECC算法正是利用该特点进行加密，点Q为公钥，大数k为私钥，点P为基点。

### 理解签名和验签
为了帮助理解签名和验证存在的动机，我们想象下面的场景。你想证明你是一个优秀的弓箭手，有能力在 500 码 （约 457 米）内，射中任何选定物体。现在，如果有人可以观察到你，并与你互动，那么要证明你的能力并不困难。他可以把你的儿子放置在 400 码的距离，头顶一个苹果，让你用箭射中苹果。作为一个优秀的弓箭手，你可以完成这个挑战来证明你的能力。验证者选定的靶使得你的箭术很容易被验证。

不幸的是，这种方法缺乏扩展性。比如你想证明给 10 个人看，你需要完成 10 次不同的挑战，射出 10 支箭，射向 10 个不同的靶。你可以让 10 个人围观你射一箭，但是因为不可能让 10 个人都指定靶，所以他们总会怀疑你是否只擅长特定的靶，而非任意的靶。我们希望，你只完成一次，且不需要和观察者交流，但他们仍然确信你是一个优秀的弓箭手，可以射中任意的靶。

比如，你简单地射中了你选定的目标，人们观测后并没有信服。毕竟有可能是你先射箭后画靶。那么你应该怎么做呢？

你可选择一种非常机智的方法。在箭的箭头上雕刻你要射中的靶的坐标（孩子头上的苹果），之后用箭射中你的靶。现在任何看到靶的人都可以使用 X 光机看看嵌入的箭头的坐标是否是靶的位置。很明显，必须在箭头射入靶之前就雕刻好坐标。所以他们可以相信你是一个优秀的弓箭手（假设这个靶并不是你反复多次练习过的）。

签名和验证使用的是相同的技巧，我们要证明的东西从优秀的箭术变为掌握一个隐秘的数字。我们希望证明我们掌握这个数字但不透露这个数字本身。这可以通过把“靶”放进计算过程，然后击中“靶”来实现

### 椭圆曲线上的签名和验签
雕刻靶坐标依赖椭圆曲线数字签名算法（Elliptic Curve Digital Signature Algorithm，ECDSA），算法需要保护的秘密e满足下面的等式：eG = P, P是公钥，e是私钥。

我们选定的靶是一个 256 位的随机数k，并且有：kG = R

于是R替换成为新的靶，实际上我们只关心R的x轴坐标，将其命名为r。

我们宣称下面的方程等价于离散对数问题：

$ uG + vP = kG $

k是随机选取的，u和v由签名者提供且均不等于 0，G和P是已知的。这个命题成立是因为：

$ uG + vP = kG⇒vP = (k–u)G $

由于v≠ 0，所以可以通过除以标量乘法的系数v得到：P = ((k–u)/v)G  
如果e已知，则有：eG = ((k–u)/v)G或者e = (k–u)/v这意味着对于任何（u，v）的组合，只要满足上述方程，将足以证明其为e持有者。如果我们不知道e，则不得不穷举（u，v）直到e = (k–u)/v。如果我们能提供满足方程的任意（u，v）组合，这意味着我们在只知道P和G的情况下，已经解决了P = eG的离散对数问题。换句话说，我们破解了离散对数问题

在签名和验证的语境下， 这被称为签名哈希（signature hash）。 哈希函数（hash）是确定性函数， 接受任意数据并转化为定长。签名函数是包含射击者意图的消息指纹，任何验证消息的人都会接收它，用字母z表示。可通过如下的方式把它雕刻在 uG+vP 的计算中。

$ u = z/s, v = r/s $

我们知道r参与了v的计算，所以箭头已经雕刻了目标。通过u的计算，我们也雕刻了射击的目的。所以射击的靶和射击的原因都已经在方程中了。

为了使方程成立，接下来计算s：

$ uG + vP = R = kG \\
uG + veG = kG\\
u + ve = k \\
z/s + re/s = k\\
(z + re)/s = k\\
s = (z + re)/k $

这就是基础的签名算法，签名的数字是r和s。

验证过程也非常直接明了：



$ uG + vP，其中 e u, v ≠ 0 \\
uG + vP = (z/s)G + (re/s)G = ((z + re)/s)G = ((z + re)/((z + re)/k))G = kG = (r, y) $

### secp256k1算法
secp256k1是指比特币中使用的ECDSA(椭圆曲线数字签名算法)曲线的参数，并且在高效密码学标准（Certicom Research，[http://www.secg.org/sec2-v2.pdf）中进行了定义。](http://www.secg.org/sec2-v2.pdf）中进行了定义。)

在比特币使用的曲线secp256k1中使用的参数如下：

+ a = 0, b=7, 曲线为$ y^2 = x^3 + 7 $
+ $ p = 2^256 - 2^32 - 977 $
+ Gx = 0x79BE667EF9DCBBAC55A06295CE870B07029BFCDB2DCE28D959F2815B16F81798
+ Gy = 0x483ADA7726A3C4655DA4FBFC0E1108A8FD17B448A68554199C47D08FFB10D4B8
+ n = 0xFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFFEBAAEDCE6AF48A03BBFD25E8CD0364141

其中，Gx, Gy 为G点的x坐标和y坐标， a,b 为选用的曲线参数，p为选取的一个非常接近$ 2^{256} $的素数，这意味着大部分小于$ 2^{256} $的素数都在这个素数域内。



### secp256k1基本流程
#### 签名流程
1. 我们已知z和满足eG=P的e。
2. 随机选取k。
3. 计算R=kG,及其x轴坐标r。
4. 计算 s=(z+re)/k。
5. (r,s) 即为签名结果。

#### 验证流程
1. 接收签名者提供的(r,s)作为签名，z是被签名的内容的哈希值。P是签名者的公钥（或者公开的点）。
2. 计算 u=z/s 和 v=r/s。
3. 计算 uG + vP = R。
4. 如果R的x轴坐标等于r，则签名是有效的

+ **大数运算提示**：在 Go 语言中，`math/big.Int` 类型**不能**直接使用 `+`、`-`、`*`、`/` 等常规符号。必须使用指针方法。例如计算 `c = (a + b) mod N` 应写成 `c.Add(a, b).Mod(c, N)`；更多基础操作请查阅 [https://pkg.go.dev/math/big#Int](https://pkg.go.dev/math/big#Int)。
+ **哈希与求模**：哈希函数在 secp256k1 中选用双 sha256，可以通过 `crypto.Keccak256([]byte)` 来实现。所有在有限域上的最终操作之后，均需要使用 `.Mod(result, N)` 进行求余运算。

# 实验二 区块链的基本结构
## 实验目的
+ 了解区块链上的简单数据结构
+ 实现 Merkle 树的构建
+ 理解以太坊交易的验证规则

## 实验介绍
### 区块链
区块链是一种分布式数字账本，通过“区块按哈希前后相连”的方式形成一条不断增长的链。每个新区块的区块头都会保存 **PrevBlockHash（上一区块哈希）**，从而把全链组织成严格的顺序结构；若链上任意区块被篡改，其后的所有区块在校验 **PrevBlockHash** 时都会失败。

在本实验代码中：

+ **NewBlockchain**：初始化区块链并创建 **创世块（Genesis Block）**；创世块会立刻交由共识引擎封装（补全 `Hash / Validator / Nonce` 等共识相关字段）。
+ **MineBlock**：从交易池取出交易，先在克隆状态上执行与验证，再打包生成新区块，并交由共识引擎完成共识字段构造。

---

### 区块
区块是区块链的基本单元，用于存储交易等信息。一个区块由 **区块头（Header）** 与 **区块体（Body）** 组成：

+ **Header**：唯一标识区块，并承载校验所需字段
+ **Body**：存放交易列表

本实验中区块结构（简化版）为：

```go
type Block struct {
    Header *BlockHeader
    Body   *BlockBody
}

type BlockHeader struct {
    Height        int64
    Hash          string
    PrevBlockHash string
    Timestamp     int64
    TxMerkleRoot  string
    AcMerkleRoot  string
    Validator     common.Address
    Nonce         int
}

type BlockBody struct {
    Transactions []*types.Transaction
}
```

字段含义（实验用的关键点）：

+ **Hash**：当前区块头哈希，可作为区块唯一索引。
+ **PrevBlockHash**：上一个区块头哈希，用于把区块链成链。
+ **Timestamp**：区块时间戳（验块时要求递增）。
+ **TxMerkleRoot**：区块内交易的 Merkle 根，用于证明“某笔交易被包含”。
+ **AcMerkleRoot**：账户状态的 Merkle 根（本实验用“账户序列构建的 Merkle Tree”近似以太坊状态树）。
+ **Validator**：共识中的出块节点/提议者。
+ **Nonce / Difficulty**：在 PoW 中通过不断试算 `Nonce` 寻找满足难度目标的区块哈希；在 PoS 中不依赖 PoW 式挖矿难度（实验里共识引擎会按各自规则使用这些字段）。

补充（对齐真实以太坊概念）：以太坊区块头中通常还会包含交易树、回执树、状态树等多棵结构的根哈希，用于为交易与状态提供可验证性。

---

### 账户
区块链可视作“有限状态机”，世界状态就是状态机的状态。以太坊账户分为外部账户与合约账户：

+ **外部账户（EOA）**：由用户私钥控制，可发起交易（转账或调用合约）。交易通过私钥签名证明“由谁发起”。EOA 通常包含地址、余额、`nonce` 等字段；其中 `nonce` 用于防止重放，并约束交易顺序。
+ **合约账户（Contract Account）**：由链上合约代码控制，不存在私钥。合约账户无法主动发起交易，只能在被外部交易或其他合约调用时执行代码，从而改变状态。合约账户除地址、余额外，还包含合约代码与持久化存储（storage）。

本实验为降低复杂性：**仅保留外部账户**，不实现合约账户与 EVM。

```go
type Account struct {
    nonce   int64
    Address common.Address
    balance float64
}

```

### 交易
交易用于驱动状态转移。以“转账交易”为例，执行会带来三类状态变化：

1. 从发送方账户扣款
2. 向接收方账户加款
3. 发送方 **Nonce** 递增（约束顺序、防重放）

为简化起见，本实验仅保留与转账及签名验证直接相关字段：

+ **From / To**：转出与转入地址
+ **Amount**：转账金额
+ **Timestamp**：交易时间戳
+ **PublicKey**：发送方公钥（用于与 `From` 关联并进行验签）
+ **Signature**：签名

交易哈希与签名/验签：

+ **交易哈希（ID）**：将 `From、To、Amount、Timestamp、PublicKey` 等字段拼接后做 **sha256**，用于唯一标识交易并防篡改。
+ **签名/验签**：采用 **ECC** 椭圆曲线签名；签名与验证前都会先对消息做哈希，再生成签名与校验合法性。

交易验证与生命周期（实验流程）：

1. 用户创建交易并使用私钥签名，将交易发送到节点。
2. 节点调用 **ValidateTransaction** 校验单笔交易：地址合法、金额 > 0、公钥与 `From` 匹配、验签通过、交易哈希一致等。
3. 校验通过后，交易进入交易池等待打包。
4. 节点出块时调用 **ValidateTransactions**：在给定状态上按顺序批量验证并“预执行”，逐笔更新余额与 `nonce`；若出现余额不足或规则不满足，立刻返回错误并拒绝打包。
5. 交易被打包进区块并广播。验证节点收到区块后会 **重新执行** 区块内交易，并将执行得到的账户状态根与区块头中的 **AcMerkleRoot** 对比；一致且共识校验通过则接收该区块。

---

### Merkle树
以太坊中正式实现会用 **MPT（Merkle Patricia Trie）** 来分别组织交易、回执与账户状态。MPT 将“Merkle 可验证性”与“字典树按键高效查找/更新”结合，但实现复杂、维护成本高。为把核心思想讲清楚并降低系统复杂度，本实验采用更直观的 **Merkle Tree**：

+ 将 **单笔交易**（或 **单个账户序列化结果**）作为叶子节点
+ 自底向上逐层哈希汇总，最终得到 **Merkle 根（Root）**
+ 用 Root 表示“这组数据的指纹”，用于包含性/一致性验证

**SPV（Simplified Payment Verification）**：轻节点不保存完整区块数据，只保存区块头（含 Merkle 根）。当轻节点要验证“某笔交易是否被某区块收录”时，全节点可提供一条 **SPV 路径（Merkle proof）**：从目标叶子到根路径上每一层所需的“兄弟节点哈希”。轻节点据此逐层复算并最终匹配区块头 Root，即可完成验证。

Merkle Tree 的构建过程（自底向上）：

1）为每个叶子计算哈希（本实验用单次 **sha256**）。  
2）若叶子数量为奇数，复制最后一个叶子补齐，保证两两配对。**（提示：此奇偶对齐逻辑要求学生在 `NewMerkleTree` 内部实现时自行去判断切片长度并处理 `append` 补齐，不要指望上层区块外部调用者已提前传好偶数给底层哈希树）。**  
3）将相邻左右子节点哈希拼接后再哈希得到父节点；重复合并直到只剩 1 个节点，其哈希即 **Merkle 根**。

Merkle 根用于两类关键验证：

+ **全节点验块**：用区块内数据重算 Root，与区块头 Root 比对。
+ **轻节点验证**：用区块头 Root + SPV 路径完成“包含性验证”，无需下载全部交易。

# 实验三 PoW与PoS共识
## 实验目的
+ 了解并实现基于PoW的共识算法出块
+ 了解并实现基于PoS的共识算法出块

## 实验介绍
### 区块链共识协议
区块链的“共识”要解决的核心问题是：在一个没有中心机构、节点可能延迟或作恶的网络里，大家如何对**下一块由谁来写**、以及**哪条链代表系统当前的有效历史**达成一致。  
在公有链里，这通常还会和激励机制绑定：成功产出并被网络接受的区块，出块者往往能获得区块奖励与手续费（常被统称为“挖矿/出块奖励”）。

以经典的 PoW 公有链为例，共识既决定“谁能写入区块”，也决定“出现分叉时选哪条链”。通用做法是用**累计工作量最大（也常被口头称为最长链）的那条链**作为主链：因为它代表了全网投入的计算成本最多，篡改它需要付出同量级甚至更大的成本。为了让出块速度保持相对稳定，系统会周期性进行**难度调整**，根据过去一段时间的出块快慢上调或下调难度，使平均出块间隔回到目标值。

### 工作量证明（POW）
工作量的证明机制，简单来说就是通过提交一个容易检测，但是难以计算的结果，来证明节点做过一定量的工作。对应的算法需要有两个特点：计算是一件复杂的事情，但是证明结果的正确与否是相对简单的。对应地行为，可以类比生活中考驾照、获取毕业证等。

工作量证明由Cynthia Dwork 和Moni Naor 1993年在学术论文中首次提出。而工作量证明（POW）这个名词，则是在1999年 Markus Jakobsson 和Ari Juels的文章中才被真正提出。在发明之初，POW主要是为了抵抗邮件的拒绝服务攻击和垃圾邮件网关滥用，用来进行垃圾邮件的过滤使用。POW要求发起者进行一定量的运算，消耗计算机一定的时间。

 中本聪将 PoW 共识引入区块链出块流程，并结合“最长链”的原则，使得只要作恶者无法掌握全网超过 50% 的算力，就难以持续改写链上历史、恶意改变当前区块链状态。在区块链中的实现通常包含这些要素：  

+ **哈希谜题（Hash Puzzle）**：把区块头（或其简化拼接）输入哈希函数，要求输出满足某个目标（最常见是“前缀零数量达到难度要求”）。矿工不断尝试不同 **Nonce**（以及可能的额外字段）去碰撞满足条件的哈希。
+ **难度调整（Difficulty Adjustment）**：系统按固定窗口重新计算难度，让平均出块时间稳定在目标值附近（例如按约每两周、每 2016 个块调整一次的设计非常典型）。
+ **分叉与链选择**：网络传播延迟会导致同一高度出现多个合法区块（短暂分叉）。节点通常选择**累计工作量更大**的分支继续构建，最终全网收敛。
+ **工程形态**：现实网络里矿工多通过矿池协作分摊收益与波动；硬件会逐步演化（CPU→GPU→ASIC）以追求单位能耗的哈希效率；系统安全性依赖“作恶成本高于收益”的经济约束。

以太坊在 PoW 时代还引入了“叔父块/ommer”奖励机制：同高度但未成为主链的合法区块也能获得部分奖励，以缓解传播延迟带来的中心化倾向并鼓励更多节点参与出块竞争。



以太坊在此基础上，降低了出块时间，并考虑到了同一高度其他非最长链的合法出块也可以作为叔父区块获得一定奖励，以此鼓励节点计算难题出块，增加了整个系统的去中心化。

我们在本实验中实现了一种简单的POW共识：

具体规则如下：

1. 固定出块难度：**区块哈希在转化为十六进制字符串（Hex String）表示时，前导字符 `'0'` 的个数必须大于 3（即至少 4 个 `'0'`，例如 `"0000a1b2..."` 算达标。请勿将其与二进制比特位长度混淆，这等同于 2 个十六进制零字节 `0x00, 0x00`）。**

2. 挖矿哈希：先基于区块头（不含 Hash 与 Nonce）计算基础哈希，然后与 nonce 拼接后做双 SHA256。

3. 挖矿过程：不断递增 nonce，直到哈希满足前缀零要求。

4. 验块过程：重新计算哈希并校验前缀零，同时检查前一区块哈希与时间戳递增。



### 


质押证明（POS）  
PoW 需要全网持续进行大规模哈希试算，其中绝大多数尝试不会成为有效区块，能源消耗与硬件竞赛是长期争议点。PoS 的核心改动是：把“获得出块权”的门槛从**算力投入**换成**资金质押**与协议惩罚，用经济担保来约束行为，从而显著减少无效计算。

以太坊已在 **2022 年 9 月 15 日**完成从 PoW 向 PoS 的切换（The Merge），用 PoS 共识层接管出块与最终性

 PoS（以太坊为代表）通常包含这些机制：

+ **验证者与质押**：参与者锁定质押资金成为验证者；被选中者负责提议区块，其他验证者对区块进行投票/证明（attestation）。
+ **随机选择与权重**：出块权通常按质押权重概率抽取，并配合随机数机制降低可预测与可操纵性。
+ **惩罚与削减（Slashing）**：若验证者双签、围攻、长期离线等，协议可扣罚其质押；严重作恶会被削减并逐出，从经济上抑制作恶收益。
+ **分叉选择与最终性（Finality）**：PoS 往往同时包含“分叉选择规则”（决定眼下跟哪条链走）与“最终性规则”（让一段历史在经济意义上不可逆），从而提升安全性与确定性。

实际算法虽然较为复杂，但核心思想很清晰：**质押越多的节点，获得出块机会的概率越高**，从而有机会获得出块奖励；而当出块节点作恶或未能按时履职时，共识机制能够识别异常，并对其质押资金进行惩罚（扣罚/削减），将相应损失用于协议规则下的再分配，以约束参与者行为并维护系统安全。

我们在本实验中实现一种简单的POS共识，具体流程与机制如下：

1. 确定性生成seed：调用 `generateSeed` 根据前一个区块的哈希 (`prevHash`) 和当前的时间槽 (`slot`)，确定性地计算产生一个本周期特定的防欺诈随机种子。
2. 随机洗牌：调用 `shuffleValidators` 利用生成的 `seed`，建立一种可验证、可重放的打散规律，对当前所有的验证者进行确定性洗牌。
3. 阈值采样选出 proposer：调用 `pickProposer` 遍历洗牌后的验证者列表。**抽选算法提示**：可将 `seed` 与当前的验证者序列（或内容标识）合并计算 Hash 并转为 `uint64` 整数。随后计算该整数是否落入与该验证者所持金部分所映射的阈值界限内 `(MaxUint64 / TotalStake * ValidatorStake)`，满足落入边界且率先达标的即成为合法的出块者（Proposer）。以此体现“持币多即抽中率高”的核心抽选方案。
4. 共识封装区块：调用 `ConstructConsensus` 组合上述三步，通过伪随机机制决定出块权柄，完成区块头的组装并记录到链上。

# 简易以太坊实验系统 - 使用文档

> 当完成以上实验后，本系统可以正常运转，下面讲述实验系统的构建与使用
## 一、 构建项目

项目依靠 Go 语言进行编写，需要本地配置好 Go 环境。

打开终端，进入项目根目录：
```bash
# 编译整个系统并生成可执行文件
go build -o simple_eth main.go
```
执行完毕后，会在当前目录下生成 `simple_eth`（在 Windows 环境下为 `simple_eth.exe`）可执行程序。

---

## 二、 启动与初始化网络

直接运行刚才生成的程序进入控制台交互界面：
```bash
./simple_eth
```

进入 `> ` 控制台后，首先应该初始化一条链条。你可以选择使用 **PoW** (工作量证明) 或 **PoS** (权益证明) 作为该链的核心共识引擎。

```bash
# 初始化基于 PoW 共识模式的区块链，并自动生成 100 个内置账户（每人初始 10000 余额）
> init pow

# 或者是使用 PoS
> init pos
```
*提示：网络必须先执行 `init` 才能调用后续其他逻辑。*

---

## 三、 查看账户与状态

网络初始化自带了一批账户资源。所有的地址转账都必须依赖这些已知地址。

```bash
# 列出系统中所有已知地址的余额和交易 nonce（包括创世预设账户以及通过转移新生成的账户）
> accounts
```
注意观察列表中打印出的十六进制地址（以 `0x` 开头），后续的发起转账动作将需要用到。

---

## 四、 发起交易 (Send)

`send` 命令将交易发送进入区块链内存池（Tx Pool）。请注意，此时交易仅在本地缓存完成签名与沙盒检查，**还并未被真正写在区块（链）上，也并未完成余额的最终扣款！**

### 1. 自动高频造数据发送：
如果你只是想快速测试出块过程，可以使用极其方便的自动打款指令。系统会随机挑选合法余额账户打包 10 笔互相交易。
```bash
> send auto
```

### 2. 手动指定打款发送：
```bash
# 用法: send <发送方 0x...> <接收方 0x...> <金额>
> send 0x7E1Fc7896419FF2C80f5F0F2923b3052428F7B14 0x3246Cc59B183f127A5D0145BE4dC96268086cF42 100
```
*提示：可以利用分号 `;` 分割来一键发送多笔交易。*

---

## 五、 矿工出块与记账 (Mine)

交易池积压后，我们需要调用共识出块。

```bash
# 激活挖矿动作
> mine
```
- **在 PoW 模式下**，系统会经历明显的“无脑哈希穷举碰撞”停顿，当找到满足难度要求的 Nonce 后，控制台将输出新区块已被产出的信息及 Merkle根、Hash 值等。
- **在 PoS 模式下**，系统则会通过预设种子在质押抽选者中伪随机敲定并直接打包这批交易。

*此时再使用 `accounts` 查询刚才涉及的账户，你会发现真实余额才被正式更新。*

---

## 六、 进阶指令测试

针对进阶与深入测试的部分，可以随时在终端交互使用以下验证指令。

### 1. 修改 PoW 动态难度
只对 PoW 模式生效，要求系统寻找前缀包含更多个 `0` 的哈希值（起步最低难度为 4）：
```bash
# 用法: changePowdifficult <连续零的数量>
> changePowdifficult 5
```

### 2. PoS 添加额外金钱质押权柄
只对 PoS 共识生效（要求金额有效且不超过 32）：
```bash
# 用法: pos-add <要注册验证者的地址> <抵押额度>
> pos-add 0x123... 15.5
```

### 3. 轻节点 SPV 默克尔树路径验证
证明某比处于链上的老交易是否存在，而不必下载整条链：
```bash
# 根据成功交易记录的 TxHash 让核心节点构建一条到达块根的 SPV 证据路径
> spv-proof <交易的哈希值TxHash>

# 轻节点或用户利用获取到的 JSON 格式证据自己完成路径推演，查验真伪
> spv-verify <JSON字符串格式的数据>
```
